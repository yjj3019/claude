# FEF v1.1.1 Release Notes

Release date: 2026-07-10

Tag: `v1.1.1`

## Summary

FEF v1.1.1 is a focused transfer-and-evaluation release.

It adds Fable5 distillation assets, S-Core OSSLab technical blog guidance, and the first coding-focused golden tests. The release measures observable Fable5 coding behaviors against Opus+FEF and Sonnet+FEF, without claiming personality imitation or hidden-reasoning transfer.

The main result: on the current deterministic coding fixtures, Opus+FEF reached Fable5 parity, while Sonnet+FEF reached near-parity with one execution-verification weakness exposed by GT014.

## Scope

This release keeps the existing architecture intact.

No new permanent layer was added.

The work is intentionally small:

- add transfer assets
- add coding tests
- record measured results
- promote only the most proven review/evidence behaviors
- avoid creating a broad Coding module until the evidence justifies it

GT014 exposed a Sonnet execution-verification failure where a correct patch landed in a redirected worktree instead of the assigned grading directory.

The GT014 prompt and runner protocol now require assigned-directory diff/test verification before reporting success.

## Evaluation Results

### GT012: Shared Root-Cause Fix

| Variant | Average | Std Dev | Result |
|---|---:|---:|---|
| Fable5 | 100 | 0 | PASS |
| Opus+FEF | 100 | 0 | PASS |
| Sonnet+FEF | 98 | 2.45 | PASS |

Finding:

- Opus matched Fable5.
- Sonnet fixed correctly but had incomplete self-reporting in some runs.

### GT013: Existing Helper Reuse

| Variant | Average | Std Dev | Result |
|---|---:|---:|---|
| Fable5 | 100 | 0 | PASS |
| Opus+FEF | 100 | 0 | PASS |
| Sonnet+FEF | 100 | 0 | PASS |

Finding:

- All variants found and reused the existing helper.
- This test is now saturated and should be treated as a regression test.

### GT014: Boundary Root-Cause Fix

| Variant | Average | Std Dev | Result |
|---|---:|---:|---|
| Fable5 | 100 | 0 | PASS |
| Opus+FEF | 100 | 0 | PASS |
| Sonnet+FEF | 96 | 8 | HOLD |

Finding:

- Sonnet's root-cause reasoning and patch shape were correct in all runs.
- One Sonnet run reported success from a redirected worktree while the assigned directory still failed tests.
- This was classified as an execution-verification/tooling failure, not a reasoning failure.

## Interpretation

Current coding-test conclusion:

- Opus+FEF reached measured parity with Fable5 on GT012-GT014.
- Sonnet+FEF reached near-parity, with execution-state verification as the main observed weakness.
- GT012 and GT013 are saturated and should not be used as high-discrimination benchmarks.
- GT014 remains useful because it caught a real process failure.

## Known Limits

- The coding fixtures are small and deterministic.
- Results do not prove universal coding superiority or broad real-world performance.
- Baseline non-FEF coding runs were not recorded for GT012-GT014.
- GT014 Sonnet HOLD depends partly on the local tool/worktree behavior observed during this session.
- Fable5 self-report is not treated as hidden-reasoning access; only observable outputs and verification results are used.

## Recommended Next Direction

Do not create a broad Coding module yet.

Recommended next test only if more discrimination is needed:

- GT015 ??Coding Contract Preservation

Suggested GT015 shape:

- an obvious patch makes new tests pass but breaks an existing documented behavior
- the fixture includes an existing test or README contract that must remain true
- scoring checks whether the model preserves both the bug fix and prior contract

## What v1.1.1 Means

v1.1.1 closes the first Fable5-to-Opus/Sonnet transfer cycle.

It provides evidence that the framework can transfer measurable coding-review behaviors in small controlled tasks, and it identifies where Sonnet needs stronger execution verification.

It does not justify broad new prompt layers. The next improvement should be data-driven, not architecture-driven.
